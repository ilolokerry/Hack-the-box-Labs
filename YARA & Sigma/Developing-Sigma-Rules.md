# Developing Sigma Rules

 
## Objective
 
This section was about learning how to build Sigma rules manually, from scratch, by working backward from real attack telemetry. The goal was to take a malicious event, figure out exactly what makes it detectable, and translate that into a structured Sigma rule — then use `sigmac` to turn that rule into a usable query for a SIEM or log management tool.
 
## Lab Setup
 
- Target system spawned via HTB Academy
- Sample: `shell.exe`, a renamed copy of Mimikatz, located at `C:\Samples\YARASigma`
- Captured Sysmon events saved as `lab_events.evtx` in `C:\Events\YARASigma`
- `sigmac` tool located at `C:\Tools\sigma-0.21\tools`
- Pre-built rules referenced from `C:\Rules\sigma`
## Walkthrough
 
### Example 1 — Detecting LSASS Credential Dumping
 
I started by running `shell.exe` and letting it perform `sekurlsa::logonpasswords`, which is the classic Mimikatz technique for dumping cached credentials straight out of LSASS memory.
 
While that ran, Sysmon Event ID 10 (ProcessAccess) picked up the activity in the background. The two fields that mattered most here were:
 
- **TargetImage** — confirms `lsass.exe` was the process being accessed
- **GrantedAccess** — the access rights requested, shown as a hex flag
[SCREENSHOT: Sysmon Event ID 10 entry in Event Viewer showing TargetImage `lsass.exe` and GrantedAccess `0x1010`]
 
The flag I needed to understand was `0x1010`. Breaking it down:
 
- `0x0010` = `PROCESS_VM_READ`
- `0x0400` = `PROCESS_QUERY_INFORMATION`
Combined, `0x1010` means a process is asking to read LSASS's virtual memory and query info about it — exactly the access pattern used in credential dumping. `0x0410` is the more "textbook" flag for this, but `0x1010` shows up just as often in real dumping tools, so it's worth detecting too.
 
From that, I built the first version of the rule:
 
```yaml
title: LSASS Access with rare GrantedAccess flag
status: experimental
description: This rule will detect when a process tries to access LSASS memory with suspicious access flag 0x1010
date: 2023/07/08
tags:
  - attack.credential_access
  - attack.t1003.001
logsource:
  category: process_access
  product: windows
detection:
  selection:
    TargetImage|endswith: '\lsass.exe'
    GrantedAccess|endswith: '0x1010'
  condition: selection
```
 
**Breaking it down field by field:**
 
| Field | What it does |
|---|---|
| `logsource` | Tells Sigma this rule applies to Windows `process_access` events — Sysmon Event ID 10 |
| `selection.TargetImage` | Matches any path ending in `\lsass.exe` |
| `selection.GrantedAccess` | Matches the `0x1010` access flag |
| `condition` | Both fields in `selection` must be true to fire |
 
This rule is saved on the target as `proc_access_win_lsass_access.yml`.
 
**Converting it with sigmac:**
 
I used `sigmac` to translate the rule into a PowerShell `Get-WinEvent` query — useful for environments without a full SIEM:
 
```powershell
PS C:\Tools\sigma-0.21\tools> python sigmac -t powershell 'C:\Rules\sigma\proc_access_win_lsass_access.yml'
```
 
[SCREENSHOT: terminal output of `sigmac` generating the PowerShell `Get-WinEvent` query]
 
This generated a ready-to-run PowerShell filter, which I then pointed at the actual `.evtx` file to confirm detection:
 
```powershell
PS C:\Tools\sigma-0.21\tools> Get-WinEvent -Path C:\Events\YARASigma\lab_events.evtx | where {($_.ID -eq "10" -and $_.message -match "TargetImage.*.*\\lsass.exe" -and $_.message -match "GrantedAccess.*.*0x1010") } | select TimeCreated,Id,RecordId,ProcessId,MachineName,Message
```
 
[SCREENSHOT: PowerShell output confirming the Sysmon Event ID 10 match against `lab_events.evtx`]
 
It successfully flagged the exact Sysmon event generated by `shell.exe` accessing `lsass.exe` — confirming the rule logic translated correctly and actually catches the behavior in a real log file.
 
**Cutting down false positives:**
 
A bare "LSASS access with this flag" rule is going to be noisy in any real environment — plenty of legitimate AV, EDR, and update tools touch LSASS. So the module pushed further into a more production-ready version that adds two layers of context:
 
1. **Expanded GrantedAccess list** — instead of just `0x1010`, the robust rule checks for a whole set of suffixes (`10`, `30`, `50`, `70`, `90`, `B0`, `D0`, `F0`, etc.) known to show up across different LSASS-dumping tools.
2. **SourceImage path filtering** — flags it as more suspicious if the *source* process is launching from `\Temp\`, `\Users\Public\`, `\PerfLogs\`, `\AppData\`, or `\htb\` — common staging directories for malware.
On top of that, the rule adds explicit `filter_optional_*` blocks to whitelist known-legitimate noisy processes (Dropbox updater, VS Code, Chrome updater, Adobe ARM helper, etc.) so the condition becomes:
 
```yaml
condition: selection and not 1 of filter_optional_*
```
 
This is the part that actually made the lesson click for me — a good detection rule isn't just "what does the attack look like," it's "what does the attack look like *that legitimate software doesn't also look like*." Tuning out known-good behavior is just as important as writing the initial detection logic.
 
### Example 2 — Multiple Failed Logins From a Single Source (Event ID 4776)
 
The second example moved away from process memory access and into authentication abuse — specifically, NTLM credential validation failures (Event ID 4776).
 
A few things worth noting about 4776 that I hadn't internalized before this lab:
- It only fires on whichever machine is authoritative for the credentials (domain controller for domain accounts, the local machine for local accounts)
- It only logs the **source workstation name** — not the destination machine
- A non-zero `Error Code` means the validation attempt failed
Using the sample log `lab_events_2.evtx`, I could see multiple failed validation attempts against a `NOUSER` account, all originating from the same `Workstation` field (`FS01`) — a pattern that looks like either a brute-force or password-spray attempt being tested against an environment.
 
[SCREENSHOT: Event Viewer showing Event ID 4776 entries with TargetUserName `NOUSER` and Workstation `FS01`]
 
The rule used to catch this pattern:
 
```yaml
title: Failed NTLM Logins with Different Accounts from Single Source System
id: 6309ffc4-8fa2-47cf-96b8-a2f72e58e538
status: unsupported
description: Detects suspicious failed logins with different user accounts from a single source system
logsource:
  product: windows
  service: security
detection:
  selection2:
    EventID: 4776
    TargetUserName: '*'
    Workstation: '*'
  condition: selection2 | count(TargetUserName) by Workstation > 3
```
 
This was my first time seeing Sigma's aggregation syntax in action: `count(TargetUserName) by Workstation > 3`. Instead of matching individual events, the condition groups events by `Workstation` and only fires once that workstation has generated **more than 3** distinct failed-login events. That's a meaningfully different detection pattern than the LSASS rule — it's threshold-based rather than single-event-based, which makes sense for catching brute-force/spray behavior instead of a one-shot malicious action.
 
Documented false positives for this rule are realistic and worth remembering for tuning: terminal servers, jump servers, Citrix-style multi-user systems, and shared workstations where many users authenticate from one source legitimately.
 
## Key Takeaways
 
- Writing a Sigma rule starts with understanding the *telemetry*, not the syntax — I had to understand what `GrantedAccess: 0x1010` and Event ID 4776 actually represent before the YAML made sense.
- `sigmac` bridges the gap between a generic Sigma rule and an actual query you can run against your own logs/SIEM — translating to PowerShell, Splunk, Elastic, etc. without rewriting logic by hand.
- Aggregation conditions (`count(...) by ... > N`) are essential for behavior-based detections like brute-force attempts, where no single event is malicious on its own.
- A genuinely useful detection rule almost always needs a false-positive-reduction layer (`filter_optional_*` blocks) — raw detection logic alone tends to be too noisy for production.
## Conclusion
 
This section gave me a much clearer picture of what "writing a Sigma rule" actually means in practice — it's not just YAML syntax, it's translating an understanding of attacker behavior (and how it differs from legitimate activity) into a structured, portable detection. Tying it together with `sigmac` made the rule feel less academic and more like something I could realistically drop into a SOC workflow.
